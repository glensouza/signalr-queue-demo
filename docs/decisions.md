# Decisions

One dated entry per architecture/implementation decision made along the way — what we chose, what we rejected, why. See `CLAUDE.md` § Workflow: "if an architecture decision arises that the spec doesn't cover and there's a materially better option, ask the owner; otherwise use judgment and record it here."

## 2026-07-10 — `POST /queue/call-next`, not `POST /queue/{id}/call-next`

**Context:** the original build spec wrote the call-next route as `POST /queue/{id}/call-next`, matching the shape of `POST /queue/{id}/complete`.

**Decision:** dropped the `{id}` segment — the route is `POST /queue/call-next`.

**Why:** call-next's entire job is to pick the next entry itself (oldest `Waiting`, by check-in time). The caller doesn't know which entry will be served next until the server tells them — there's no id to put in the URL. `complete` is different: the caller already knows which `Serving` entry they're finishing, so `{id}` stays there.

**Rejected:** keeping `{id}` and ignoring it server-side — adds a parameter that means nothing and invites a caller to (wrongly) assume they can pick who gets called next.

## 2026-07-10 — `EnsureCreatedAsync`, not EF Core migrations

**Decision:** `SqliteQueueRepository`'s `QueueDbContext` is stamped with `Database.EnsureCreatedAsync()` at startup (see `Program.cs`), not a migrations project.

**Why:** this is a POC scaffold with no schema history to preserve and no production deployment pipeline that needs incremental upgrades. Migrations require a design-time `IDesignTimeDbContextFactory`, a `Migrations/` folder, and `dotnet ef` tooling — all overhead with no payoff when every dev/demo run starts from a fresh (or seeded) `.db` file.

**Tradeoff, accepted on purpose:** `EnsureCreated` stamps the schema once and does nothing on later model changes — if a future change adds a column, the already-created `App_Data/queue.db` won't pick it up. The fix in that case is deleting the local `.db` file (it's git-ignored and reseeds on next run), not migrating it. If this repo ever needs to preserve real data across schema changes, that's the trigger to switch to migrations.

**Rejected:** EF Core migrations — correct for production, unnecessary ceremony for a reference POC whose database is disposable by design.

## 2026-07-10 — SQLite `busy_timeout` + WAL on every connection

**Decision:** a `QueueConnectionInterceptor` runs `PRAGMA busy_timeout = 5000` and `PRAGMA journal_mode = WAL` on every connection open (registered via `AddInterceptors` in `Program.cs`).

**Why:** the API gives each HTTP request its own scoped `DbContext`/connection, and the demo deliberately drives concurrent writes — a kiosk checking in while staff calls the next entry, across the Angular apps and Blazor at once. Without `busy_timeout`, Microsoft.Data.Sqlite fails the losing writer of any overlap immediately with "database is locked" (SQLITE_BUSY) → an unhandled HTTP 500. A short busy timeout makes the second writer wait the (sub-millisecond) time for the first to finish instead. WAL additionally lets the queue-display board's reads run without blocking the writer. Verified with a 20-request concurrent write burst: all `200`, no locks.

**Why an interceptor, not a startup call:** `busy_timeout` is per-connection (not stored in the file), so a one-time PRAGMA at startup would be silently lost on every pooled connection opened afterward.

**Rejected:** relying on the driver default (0 = fail immediately) — surfaces as intermittent 500s during exactly the live demo scenario the POC exists to show.

## 2026-07-10 — Broadcasts go through one `QueueBroadcaster` choke point, best-effort, at the endpoint layer

**Decision:** all three mutations broadcast `QueueUpdated` by calling a single injected `QueueBroadcaster.BroadcastAsync(update)` from `QueueEndpoints`, right after the successful `IQueueRepository` call returns — not from inside `SqliteQueueRepository`, and not by each handler touching `IHubContext` itself. `QueueBroadcaster` wraps `IHubContext<QueueHub, IQueueHubClient>`.

**Why the endpoint layer, not the repository:** `IQueueRepository` is deliberately storage-only (see its type doc) so the Azure Table Storage implementation stays a drop-in swap — it shouldn't need to know about SignalR. Broadcasting after the repository call also guarantees the push never fires until *after* `SaveChangesAsync` has committed, so a client that reacts by immediately calling `GET /queue/since/{seq}` always sees that change already persisted — no window where the broadcast outruns its data.

**Why one `QueueBroadcaster` and not three inline `IHubContext` calls:** the three handlers broadcast identically; a single method is the one place to change *how* broadcasting works — group/target filtering when the trust boundary lands, the Azure SignalR path, logging. The future Blazor `NotifyMutation` hub method (see "Blazor is self-encapsulated" below) calls the same broadcaster, so there's exactly one broadcast implementation regardless of what triggered the write.

**Why best-effort (a failed push never fails the caller):** the write is already committed by the time we broadcast, so `QueueBroadcaster` catches and logs a broadcast exception instead of letting it propagate. Letting it bubble would turn a successful check-in into an HTTP 500 whose caller retries — double-checking-in a visitor, or skipping a waiting person on a retried call-next. ADR-0001's "a notification must never silently drop" is honored not by failing the caller but by the catch-up protocol: the committed change is replayed via `GET /queue/since/{seq}` on the client's next reconnect.

**Known limitation, accepted on purpose:** two broadcasts from concurrent requests aren't guaranteed to arrive in sequence-number order — a check-in and a call-next landing at nearly the same instant can commit in one order but have their sends scheduled by the runtime in the other. Likewise a live `QueueUpdated` can reach a just-connected client before its `CurrentSequence` baseline, because SignalR admits a connection to the broadcast set before `OnConnectedAsync` runs. Both are made harmless the same way: `QueueHub`/`IQueueHubClient` XML docs tell clients to fold every sequence number in with `max()` — the sequence number, not arrival order, is authoritative — so `CurrentSequence` is a floor, not a reset. Serializing all broadcasts through one writer to force strict ordering would be unnecessary ceremony for a demo of dozens of concurrent connections, not thousands.

**Rejected:** a single background dispatcher/channel serializing all broadcasts — solves a problem (strict push ordering) the reconnect/catch-up protocol already solves a different, more robust way.

## 2026-07-10 — GET /queue/since/{seq} catch-up cutoff: 200 events, else a full snapshot

**Decision:** `SqliteQueueRepository.GetChangesSinceAsync` returns the raw diff of `QueueChangeEvent`s when there are 200 or fewer, and a full `QueueSnapshot` (`QueueChangesSinceResponse.IsSnapshot = true`) otherwise — same fallback for a sequence number that's negative or ahead of the latest one the server has ever issued (`MaxCatchUpEvents` in `SqliteQueueRepository`).

**Why 200:** arbitrary but generous for this POC's scale (dozens–hundreds of entries per day, per the ADR-0001 performance expectations) — a client would need to miss roughly a full day of check-in/call-next/complete activity to hit it. The point isn't the exact number, it's having *a* bound: past it, a snapshot is a single fixed-size payload instead of a diff that grows without limit the longer a client stays disconnected, and it's the same end state the client would end up computing from the diff anyway.

**Why an unrecognized sequence number gets the same treatment:** sequence numbers only start at 1 and increase, so a negative or future value can't have come from a client that legitimately talked to this server before — most likely a dev `.db` file reset since it last connected (see the `EnsureCreated` decision above). Returning an error here would force every client to special-case "my sequence number was rejected"; falling back to a snapshot means a client always gets *something* to resync from, no special-casing required.

**Rejected:** erroring (400/409) on an unrecognized sequence number — correct in spirit, but pushes the recovery logic onto every client instead of handling it once, in the one place that already knows how to build a snapshot.

## 2026-07-10 — Each mutation and the catch-up read run in one explicit transaction, for consistent reads

**Decision:** `SqliteQueueRepository` wraps each mutation (`CheckInAsync`, and the `RecordChangeAndBuildUpdateAsync` tail shared by call-next/complete) and the catch-up read (`GetChangesSinceAsync`) in a single explicit `BeginTransactionAsync`. Check-in also *counts the visitor's position after* the insert rather than before.

**Why (three concurrency bugs a code review surfaced):**
- **Position:** `position = count(Waiting)` used to run *before* the insert, in its own statement. Two simultaneous check-ins both read the same pre-insert count and were both told "you're #N". Counting *after* the insert, inside the write lock `SaveChangesAsync` already holds, makes the new entry part of the count and forces a concurrent check-in to be either already-committed-and-counted or still-blocked-on-the-lock — so positions come out distinct. Verified: 5 concurrent check-ins → positions 3,4,5,6,7, no collision.
- **Broadcast `Summary`:** the snapshot for the `QueueUpdated` payload is read *after* `SaveChangesAsync`. Without a transaction, a change committed by another request in between made the seq-N broadcast carry a `Summary` reflecting seq-N+1 state — a payload internally inconsistent with its own sequence number. Reading the snapshot while still holding the write lock pins it to exactly this change.
- **Catch-up read:** `GetChangesSinceAsync` issues `MAX`, `COUNT`, and the events fetch as separate statements. Without a transaction, a commit between them let the response's `SequenceNumber` lag the highest sequence number in its own `Changes` list. One read transaction gives all three queries a single consistent SQLite snapshot (WAL keeps a read transaction's view stable for its duration, and read transactions never block writers).

**Why this is cheap here:** it's one `BEGIN`/`COMMIT` per request over a handful of rows, riding on the WAL + `busy_timeout` setup already chosen above — a second writer waits the sub-millisecond for the first, no `SQLITE_BUSY`. Verified under a 5-way concurrent check-in burst: all `200`, no locks.

**Rejected:** leaving the reads in autocommit and documenting the races as "self-healing" — they mostly are (a client folding in `max()` recovers), but a reference implementation the vendor team copies shouldn't ship a broadcast payload that contradicts its own sequence number when making it consistent costs one transaction.

## 2026-07-10 — Blazor is self-encapsulated: shared library, not REST calls to the API

**Context:** `SignalRQueueDemo.Web` and `SignalRQueueDemo.ApiService` run as separate Aspire process resources. The build spec says Blazor should be self-encapsulated and make no calls to the API — check-in/call-next/complete reuse the same .NET code the API uses via a shared library, rather than Blazor acting as just another REST client like the Angular apps.

**Decision:** Blazor's check-in/call-next/complete pages call directly into a shared queue-service library that both `SignalRQueueDemo.ApiService` and `SignalRQueueDemo.Web` reference — no HTTP calls to the API's REST endpoints. Blazor still opens its own SignalR `HubConnection` to `QueueHub` (hosted in `ApiService`) for live updates, same as the original spec's "direct `HubConnection` since it's already .NET."

**Why:** self-encapsulation is the actual value proposition being evaluated for Blazor Server in this comparison — same-runtime code reuse with no network hop to its own sibling process, versus Angular which has no choice but to go over REST. Making Blazor call the API's REST endpoints instead would make it functionally indistinguishable from a same-origin Angular app and defeat the point of the comparison.

**Broadcast wrinkle, and how it's resolved:** `QueueUpdated` broadcasting lives in one place — `QueueBroadcaster`, invoked from `QueueEndpoints` (see "Broadcasts go through one `QueueBroadcaster` choke point" above) — specifically so `IQueueRepository` stays storage-only. Because `ApiService` and `Web` are separate processes, a Blazor-initiated write via the shared library commits to the database but doesn't reach that broadcast on its own — other connected clients (Angular apps, other Blazor sessions) would silently miss it. Fix: Blazor reuses the `HubConnection` it already holds for receiving pushes to also *send* — after a local write it invokes a hub method (e.g. `NotifyMutation(sequenceNumber)`); `QueueHub` handles that call server-side and pushes through the same `QueueBroadcaster` to every client exactly as an API-originated change would. This keeps the broadcast trigger inside SignalR rather than adding a REST call, so "no API calls" still holds.

**Rejected:** Blazor calling the REST API for mutations like Angular does — simpler to reason about, but throws away the reason to pick Blazor Server for this comparison in the first place.

**Follow-up for the Blazor work:** the queue-service/repository code (`IQueueRepository` and implementations) currently lives inside `SignalRQueueDemo.ApiService/Persistence`. Building Blazor's pages will require extracting that into a shared library both projects reference — noted here so it isn't a surprise when that work starts.

## 2026-07-10 — Table Storage: gap-free sequence via optimistic insert, position via an ETag counter, both reconciled at startup

**Context:** `SqliteQueueRepository`'s monotonic sequence number comes for free from SQLite's autoincrement rowid, generated inside the same `SaveChangesAsync` transaction as the entry mutation it belongs to (see the `EnsureCreatedAsync`/transaction decisions above); it is gap-free and, because that transaction also brackets the catch-up read, a client's catch-up baseline can never run ahead of the events it was handed. Check-in's reported position similarly relies on that transaction's write lock to count `Waiting` rows consistently. Azure Table Storage has no autoincrement and no cross-table transactions, so `TableStorageQueueRepository` needed different mechanisms — and a first cut using an ETag-incremented counter for the sequence number turned out to reintroduce exactly the anomaly SQLite's transaction prevents (see below).

**Decision — sequence numbers by optimistic insert into the change-event log, not a counter.** `AppendChangeEventAsync` allocates a number by *inserting the change-event row itself*: take the next candidate, `AddEntity` it (its zero-padded RowKey is the number), and on `409 Conflict` (a concurrent writer already took that number) advance and retry. A number therefore cannot exist until its row does, which makes the log **gap-free and strictly in-order** — event N cannot be written until N-1 exists to conflict against. A `Sequence` counter row survives only as an O(1) *start hint* so allocation needn't scan the log; it may lag the true max but never leads it, so a stale hint costs at most a couple of extra retries, never a wrong or skipped number.

**Why the counter-based first cut was wrong (the bug this replaced):** incrementing a `Sequence` counter and *then* writing the event row are two non-atomic steps. A concurrent `GetChangesSinceAsync` could read the counter at N while event N's row wasn't written yet — and, because two writers' event-row writes race, event N could even be persisted before event N-1. Either way the catch-up read could return a `SequenceNumber` ahead of (or with a gap below) the `Changes` it actually contained; a reconnecting client that folds that number in as its new baseline **permanently skips the missing event**. SQLite avoids this with its transaction; the counter did not. Optimistic-insert allocation removes the failure at the root (the log is always gap-free and in-order), and `GetChangesSinceAsync` now derives the returned `SequenceNumber` from the last event it actually returns, never from the hint — so it can never hand back a baseline ahead of the diff. Verified against a live Azurite emulator: under 20 concurrent check-ins interleaved with catch-up reads, every non-empty diff's `SequenceNumber ≤ max(seq in Changes)` and the log stayed contiguous `1..N`.

**Decision — check-in position via an ETag counter, closing the earlier position race.** Position is the running `WaitingCount` singleton row, adjusted by `AdjustCounterAsync` (read value + ETag, `UpdateEntity(ifMatch: ETag)`, retry on `412`). This replaced a query-time `COUNT(Waiting)` that two simultaneous check-ins could both read pre-insert and both report the same position; the ETag precondition now serializes concurrent check-ins into distinct positions. Verified: 5 concurrent check-ins on an empty queue → positions exactly 1–5, both via direct calls and via concurrent HTTP `POST /checkin` against the running `aspire run` stack.

**Contention bound + backoff.** All of these retry loops share `MaxOptimisticConcurrencyAttempts = 32`. The binding case is the single hot `WaitingCount` row: N truly-simultaneous check-ins each need up to N ETag retries (one winner per round), so the bound is the largest instantaneous check-in burst the position counter tolerates — 32 is comfortable headroom over this POC's handful of kiosks. Retries also jitter-back off (`BackoffAsync`) so contenders de-synchronize instead of thundering-herd re-colliding. (A 20-way burst exhausted an earlier bound of 10; 32 + backoff handles it cleanly.)

**Residual crash-window drift, and how it's healed.** `WaitingCount` (and the `Sequence` hint) can still drift if a caller commits a counter update and then crashes before writing the row it was for — a single-`await`-wide window. Rather than accept that as a standing inconsistency, `TableStorageQueueSeedData.EnsureTablesAndSeedAsync` **reconciles both counters to the real persisted state on every startup** (`WaitingCount` = actual count of `Waiting` rows, `Sequence` hint = actual max event number), single-threaded before the app serves traffic. On a normal restart it's a harmless idempotent recompute; after a crash it self-heals. The sequence *authority* is the gap-free log regardless, so a lagging hint is also self-correcting at runtime. Verified: corrupting `WaitingCount` to 999 and the `Sequence` hint to 1, then re-running startup, restored both to the real values and the next check-in reported a correct position and `maxSeq+1`.

**Same ETag pattern reused for call-next/complete, on the entry row itself:** two staff members racing to call the same next entry — the loser's `UpdateEntity` gets `412`, and retrying re-runs "find the oldest Waiting entry", which now skips the just-served row and picks the next. Verified with a 5-way concurrent `CallNextAsync` burst (direct and HTTP): all 5 succeeded, 5 distinct entries, no double-serve.

**Rejected:** (1) the ETag-counter sequence number — reintroduced the catch-up anomaly above. (2) A per-request distributed lock (e.g. a lease blob) to fully serialize writes like SQLite's file lock — defeats the point of demonstrating Table Storage (no lock infrastructure to run), and optimistic insert + the ETag position counter already give the guarantees that matter for less machinery.

## 2026-07-10 — Document metadata gets its own `IDocumentRepository`, backed by whichever store `IQueueRepository` is using

**Context:** document metadata (display filename, content type, size, uploaded-at) needs to be tracked "alongside the queue entry so the staff console can list without hitting blob storage."

**Decision:** a new `IDocumentRepository` interface, separate from `IQueueRepository`, with a `SqliteDocumentRepository` (sharing `QueueDbContext`/connection with `SqliteQueueRepository`) and a `TableStorageDocumentRepository` (a new `QueueDocuments` table, partitioned by entry id, alongside `TableStorageQueueRepository`'s three tables). Registered in `Program.cs`'s existing `Persistence:Provider` switch, one new line per case.

**Why "alongside the entry" means "same database, own interface," not "same interface":** literal — the metadata store for a given provider is the same SQLite file or the same Azurite Table Storage account the queue entry lives in, so a staff console list is one local read, never a live Blob container enumeration. It's a separate *interface* from `IQueueRepository`, though, because document metadata isn't part of the queue mutation/broadcast pipeline `IQueueRepository` exists to abstract (no sequence number, no `QueueUpdated` broadcast) — folding it in would make that interface responsible for two unrelated concerns and complicate a future backend swap that only wants to change one of them.

**Rejected:** deriving the staff document list from a live Blob Storage container listing instead of a metadata store — the requirement explicitly rules this out ("so the staff console can list without hitting blob storage"), and it would also lose the display filename/content-type/size fields Blob Storage's own listing API doesn't return without a per-blob properties fetch.

## 2026-07-10 — Blob Storage: container-per-entry, always-randomized blob names, blob write before metadata write

**Decision:** `DocumentBlobStore` creates one Blob container per queue entry (`docs-{entryId}`), lazily on first upload for that entry — not one shared container for every document, and not pre-provisioned per check-in. Every blob name is a fresh `Guid.NewGuid()` (`DocumentBlobStore.NewBlobName`), never the client-supplied filename. The upload endpoint writes the blob first, then the `IDocumentRepository` metadata row.

**Why container-per-entry:** scopes the storage boundary to the unit a future authorization check would actually reason about ("can this staff session see this visitor's documents"), so that boundary is cheap to enforce later even though this implementation itself ships no per-container ACLs — the alternative (one shared container, one metadata table doing all the filtering) would put 100% of the access-control burden on code that can be bypassed by talking to Blob Storage directly.

**Why always-randomized blob names:** a client-controlled storage path on a public, unauthenticated upload endpoint is a path-traversal and same-name-collision vector — "never trust the client filename" applies to more than just cosmetic file naming. The original filename survives only as `DocumentMetadata.FileName`, shown to staff, never used to address storage.

**Why blob-before-metadata, not metadata-before-blob or a two-phase write:** Blob Storage and the metadata store aren't written transactionally together (different services, no distributed transaction), so one order has to be "less bad" than the other. Blob-first means a metadata-write failure after a successful upload leaves an orphaned, harmless, cleanable blob nobody will ever list. Metadata-first would instead let a subsequent blob-write failure leave a metadata row promising content that was never stored — a staff member sees the document in the list, opens it, and gets a 500 (`DocumentBlobStore.OpenReadAsync` returning null is exactly this case, reported distinctly from "document doesn't exist"). The first failure mode is invisible to staff; the second isn't.

**Rejected:** a two-phase commit / saga across Blob Storage and the metadata store to make the pairing atomic — real engineering for a real DMS integration, but out of proportion to a POC whose blob-first ordering already makes the one failure mode that can occur harmless to the staff-facing experience.

## 2026-07-10 — Upload content-type allowlist checked against the client's `Content-Type` header, not file magic bytes

**Decision:** `POST /checkin/{id}/documents` rejects any upload whose reported `Content-Type` isn't `application/pdf`, `image/jpeg`, or `image/png`, checked against the header the client sends with the multipart part — not by sniffing the file's actual bytes for a PDF/JPEG/PNG signature.

**Why:** a header check is a couple of lines and catches the overwhelmingly common case (a legitimate kiosk client sending a real content type for a real file). Byte-sniffing catches a client that lies about its own header, but for this POC's threat model — a public, no-real-court-data demo endpoint, not a production DMS integration — that's added complexity (a small format-detection library or hand-rolled magic-byte table) for a risk this reference implementation would rather document honestly than half-solve.

**Documented limitation, not a silent gap:** a malicious client can still send `Content-Type: application/pdf` with, say, an executable's bytes inside; the allowlist check will pass it. This is called out at the validation site in `DocumentEndpoints.cs` and here, matching the project's "raise the bar, document what it doesn't do" theme already established for the public-endpoint hardening.

**Rejected:** magic-byte sniffing at upload time — the more correct answer for a production DMS-facing endpoint, and worth revisiting if this POC's upload path is ever adapted past reference-implementation status.

## 2026-07-10 — Document viewing endpoints ship unauthenticated; mock staff auth gates them separately

**Decision:** `GET /queue/{id}/documents` and `GET /queue/{id}/documents/{docId}` have no auth check in this change — a `TODO` comment on `DocumentEndpoints.MapDocumentEndpoints` marks them for the `X-Staff-Key` endpoint filter that the mock staff-auth work adds.

**Why:** the mock staff-auth filter is built as its own piece of work and already covers document viewing among the endpoints it gates (alongside call-next/complete) — building a second, throwaway auth mechanism here just to have *something* would mean that filter immediately replaces it, doubling the work for no benefit. These endpoints are designed to sit behind that shared filter, and the endpoint list is coordinated with it.

**Rejected:** inventing a lightweight ad hoc check (e.g. a hardcoded header) scoped to just these two endpoints — indistinguishable from real auth to a reader skimming the code, and directly contradicts CLAUDE.md's documentation standard of never leaving a security-relevant gap undocumented.

## 2026-07-11 — `POST /checkin/{id}/documents` calls `.DisableAntiforgery()`

**Context:** verifying the document upload end-to-end against a live Azurite emulator, every upload 500'd with `System.InvalidOperationException: ... contains anti-forgery metadata, but a middleware was not found that supports anti-forgery.` ASP.NET Core's minimal APIs auto-attach antiforgery-validation metadata to any endpoint binding `IFormFile`, and refuse to serve the route at all unless `app.UseAntiforgery()` (ASP.NET Core's cookie-based antiforgery system) is registered.

**Decision:** `POST /checkin/{id}/documents` calls `.DisableAntiforgery()` to opt out of that built-in system, with an inline comment at the call site explaining why.

**Why not just add `app.UseAntiforgery()` instead:** ASP.NET Core's built-in antiforgery system is cookie-based — it pairs a same-site cookie with a token, which assumes a browser session with cookies flowing back to the same origin. The public-endpoint hardening work already specifies a different, stateless mechanism for this exact endpoint: a short-lived HMAC token from `GET /checkin/token` that the kiosk fetches and echoes back on `POST`, chosen specifically because a public kiosk SPA (behind a different origin, per the CORS hardening in the same work) can't be assumed to carry a same-site cookie. Wiring up the built-in system now would mean ripping it back out for that token instead, for zero benefit in between.

**Verified:** caught by manually exercising the upload endpoint against a live Azurite container (`docker ps` showed the emulator already running from a prior `aspire run`) before this fix, and confirmed fixed by re-running the same request afterward — this is exactly the kind of gap `dotnet build` alone doesn't catch, since the failure only happens at request-routing time, not compile time.

**Rejected:** leaving `app.UseAntiforgery()` unregistered and hoping the later hardening work catches this — would have shipped the document upload with its own acceptance criterion ("Upload via `.http` file succeeds") unverified and failing.

## 2026-07-11 — Upload hardening: a buffering backstop and a per-entry document cap, on top of the per-file size check

**Context:** the per-file size check (`IFormFile.Length > 10 MB → 400`) has two gaps on a public, unauthenticated endpoint. First, `IFormFile.Length` isn't known until the framework has *already buffered the whole body*, so the check bounds what reaches Blob Storage but not how much a caller can make the server spool per request. Second, nothing bounds the *number* of uploads against a single entry, so an actor who learns one entry id (returned in plaintext by `GET /queue`) could push unlimited 10 MB blobs against it.

**Decision — a buffering backstop via `WithFormOptions`.** The upload endpoint sets `bufferBodyLengthLimit` and `multipartBodyLengthLimit` to the file cap plus 64 KB of envelope headroom. A grossly oversized body is now cut off by the form reader *before* the handler runs, instead of spooled in full. Because that path throws inside form binding (an `InvalidDataException` the framework wraps in a `BadHttpRequestException`) rather than reaching the handler's tidy check, a small `UploadLimitExceptionHandler` maps it to a clean `413 Payload Too Large` `ProblemDetails` instead of the opaque 500 it would otherwise produce. The handler's own `Length` check still runs for a file only modestly over the cap, giving the more precise "exceeds the 10 MB upload limit" message; the backstop covers the far larger bodies that never reach it.

**Decision — a soft per-entry document cap (`MaxDocumentsPerEntry = 10`).** The upload endpoint counts an entry's existing documents and returns `409 Conflict` once it is at capacity. Deliberately *soft* — checked, not transactionally enforced, so two truly simultaneous uploads could both pass it — because a hard guarantee would need a transaction across the metadata store for no real benefit at this scale, where a visitor legitimately attaches at most a handful of supporting documents. The point is a bound on the storage-exhaustion vector, not an exact invariant.

**Why these are honest, documented limits, not security theater:** the buffering backstop still lets a caller spend the ~10 MB headroom per request, and the per-entry cap is soft — both are POC-appropriate bars that raise the cost of abuse without claiming to be airtight, consistent with the rest of the public-endpoint hardening.

**Rejected:** raising the per-file cap into the buffering limit so the two never disagree — would let the clean, precise in-handler message cover more of the range but leave the buffering unbounded, which is the exact gap the backstop exists to close. Rejected a hard, transactional per-entry cap — real machinery for a bound that a soft check already delivers at this scale.

## 2026-07-11 — Mock staff auth as an `IEndpointFilter`, not middleware or an attribute

**Decision:** `StaffAuthFilter` is an `IEndpointFilter` applied per-endpoint via `.AddEndpointFilter<StaffAuthFilter>()` on `POST /queue/call-next`, `POST /queue/{id}/complete`, and the two document-viewing `GET`s — not global middleware gating routes by a path prefix, and not an MVC-style attribute (this API has no controllers to attach one to).

**Why:** the four staff routes don't share a URL prefix (`/queue/call-next` and `/queue/{id}/documents` both live under `/queue`, but so does the public `GET /queue`), so a path-prefix middleware check would need its own routing logic duplicating what endpoint routing already does. `.AddEndpointFilter<T>()` is the minimal-API-native way to attach per-route behavior, resolves `StaffAuthFilter` from DI like any other service, and reads at the mapping call site exactly which routes are gated — a reader doesn't have to cross-reference a separate middleware's route-matching rules to know what's protected.

**The comparison is fixed-time.** `CryptographicOperations.FixedTimeEquals` instead of `providedKey == expectedKey` — a naive string comparison returns as soon as it finds a mismatched character, so response time leaks how many leading characters of a guess were correct. The static demo key in this POC isn't worth attacking, but the filter is the reference pattern the vendor team copies for a real deployment, so it demonstrates the correct habit rather than a shortcut that would be a real vulnerability with a real secret.

**Rejected:** a global middleware keyed off `HttpContext.Request.Path.StartsWithSegments("/queue")` plus an explicit allowlist carve-out for `GET /queue` and `GET /queue/since/{seq}` — works, but the exception list is easy to get wrong (add a new public `/queue/...` route later, forget the carve-out, and it's silently gated) compared to the filter's opt-in-per-route shape, where a new route is public by default until someone deliberately adds the filter.

## 2026-07-11 — Check-in token is a stateless signed timestamp, not a server-tracked random value

**Context:** `GET /checkin/token` needed to issue something the check-in path could later validate on `POST /checkin` (via `CheckInTokenFilter`) and `POST /checkin/{id}/documents` (inline — see the "reads its multipart body by hand" entry below), without the built-in cookie-based antiforgery system (see the `.DisableAntiforgery()` entry above for why that was off the table).

**Decision:** the token is `{expiryUnixSeconds}.{HMACSHA256(expiryUnixSeconds)}` — `CheckInTokenService` (`Endpoints/CheckInTokenService.cs`) signs a future timestamp with a server-held key and later re-derives the expected signature for whatever timestamp the caller presents, rather than generating a random opaque value and remembering it (in memory, a cache, or a database row) until it's spent or expires.

**Why:** a server-tracked value needs somewhere to track it — in-memory breaks the moment there's more than one API instance or a restart happens mid-flow (a kiosk that fetched a token right before a deploy would fail on its next check-in for no reason visible to it), and a distributed cache or DB table is real infrastructure for what this mechanism admits, out loud, is "raise the bar against drive-by scripted abuse," not a security boundary that justifies the operational cost. A signed timestamp needs no storage at all — validity is a pure function of the token plus the current time, so it works identically across restarts and multiple instances, and there is nothing to clean up for expired tokens.

**Tradeoff, accepted on purpose:** unlike a server-tracked value, a signed timestamp can be validated more than once before it expires — there's no way to mark "this exact token was already spent." That's fine here: the goal is bounding how long a captured token stays useful, not single-use enforcement, and a public kiosk flow has no natural notion of "spent" anyway (a visitor could legitimately retry a failed check-in with the same token).

**Rejected:** a random token cached server-side with a TTL — the more conventional anti-forgery-token shape, but adds a storage dependency (even an in-memory `IMemoryCache` complicates the "works identically across restarts/instances" property above) for a POC that has no such dependency anywhere else on the check-in path.

## 2026-07-11 — CORS allowed origins are config, seeded with a placeholder (no Angular app exists yet to reference)

**Context:** restricted CORS needed a concrete allowed-origins list. The Angular apps (`public-checkin`, `queue-display`, `internal-queue`) that would actually call this API cross-origin don't exist as AppHost project resources yet — they're later work items — so there's no Aspire service-discovery-injected origin to read the way the Table/Blob Storage connection strings are.

**Decision:** `Cors:AllowedOrigins` in `appsettings.json`, read once at startup into an array passed to `AddCors`. Seeded with `http://localhost:4200`–`4202` — one fixed dev-server port per Angular app (assigned in `angular.json` under `serve.options.port`), so all three can run simultaneously against the one API during local development — as an honest placeholder ahead of containerization, not real deployment origins.

**One policy (`KnownFrontends`), applied to every browser-reachable surface — the public *and* staff REST endpoints and the SignalR hub — not just the public endpoints.** CORS is not the trust boundary here (`StaffAuthFilter`/`CheckInTokenFilter` are), so scoping it to only the public endpoints would have been a category error: it would leave the (cross-origin) staff Angular app unable to call `call-next`/`complete` at all, and — worse — leave the SignalR hub, which *every* frontend depends on for live updates, un-negotiable from any browser. An unlisted origin is still refused everywhere; a known frontend (public or staff) is let through to *then* face the real auth check. The hub carries the policy but no auth gate: it exposes no client-callable mutation and only pushes already-public queue state.

**Why config, not compiled in:** it keeps the door open for the Angular work items to update this to real container origins (or wire it through Aspire service discovery once those resources exist) without touching `Program.cs` again, matching the same "config, not compiled in" pattern already used for `Persistence:Provider`.

**Rejected:** hardcoding a fixed origin list in `Program.cs` — same operational rigidity `Persistence:Provider` already argues against, for a value even more likely to change soon (once the Angular containers exist, this placeholder is wrong). Rejected leaving CORS wide open (`AllowAnyOrigin`) until real origins exist — that's not "restricted CORS" at all, and would ship the acceptance criterion ("CORS blocks an unknown origin") failing by construction. Rejected scoping CORS to only the public endpoints (leaving staff endpoints and the hub with none) — it reads as "staff is more locked down" but actually just means the legitimate staff SPA and every frontend's live-update connection break at the browser, since CORS gates the browser, not the attacker.

## 2026-07-11 — Document upload reads its multipart body by hand, to gate the check-in token before buffering

**Context:** `POST /checkin/{id}/documents` is public and needs the same check-in token as `POST /checkin`. The obvious wiring — an `.AddEndpointFilter<CheckInTokenFilter>()` like the other check-in POST — has a subtlety: minimal-API endpoint filters run *after* parameter binding, and binding an `IFormFile` parameter reads (buffers) the whole multipart body. So a filter would let an unauthenticated caller still make the server spool an entire upload (bounded to ~10 MB by the form-options limit, but spooled all the same) before the 401.

**Decision:** the upload handler takes `HttpRequest` instead of an `IFormFile` parameter and reads the form by hand (`ReadFormAsync`), after first calling the shared `CheckInTokenFilter.Reject(...)` gate (and the entry-existence check). A request with no valid token — or for an unknown entry id — is now turned away before a single body byte is read.

**Consequences, both handled:** (1) not binding `IFormFile` means no antiforgery metadata is auto-attached, so the `.DisableAntiforgery()` call (see the entry above) is no longer needed and was removed — the app still registers no antiforgery middleware. (2) A manual `ReadFormAsync` does *not* inherit the endpoint's `.WithFormOptions(...)` metadata, so the buffering bound moves onto an explicit `FormFeature`/`FormOptions` set inside the handler (`UploadFormOptions`), preserving the exact same `MaxUploadBodyBytes` limit; an over-limit body still throws `InvalidDataException`, which `UploadLimitExceptionHandler` still maps to a clean `413`. Both were re-verified end-to-end against a live Azurite emulator (401 before buffering, 201 on success, 413 on a grossly oversized body, 400 for wrong type / missing file / non-multipart).

**Rejected:** keeping the filter and accepting the pre-auth buffering as "bounded anyway" — the bound is the *backstop's* job, not the token's; letting an unauthenticated request drive any buffering when a header check could reject it first is exactly the kind of thing this reference implementation should model correctly. Rejected a path-prefix middleware to check the token before routing — same duplicated-routing objection as the staff-auth entry above, and it would have to distinguish `POST /checkin` from `POST /checkin/{id}/documents` by hand.

## 2026-07-11 — What the Azure SignalR Emulator can and can't prove; the `UseAzureSignalR` flag stays a side-channel, not a topology switch

**Context:** ADR-0001 names a one-line escape hatch (`AddAzureSignalR(connectionString)`) for if self-hosted SignalR outgrows a single App Service instance. The [Azure SignalR Local Emulator](https://learn.microsoft.com/en-us/azure/azure-signalr/signalr-howto-emulator) is the only local stand-in, but its docs state it supports **serverless mode only** — the escape hatch ADR-0001 actually names is default (server) mode, which the emulator cannot run.

**Decision:** implement both paths, but only one of them ever executes. `AzureSignalRDefaultModeStub.Apply(services, connectionString)` is the real one-line change — it compiles, is reviewable, and is never called from `Program.cs` under any configuration, because there is no local target for it to run against. `AzureSignalRServerlessDemoService` (a hosted service, registered only when `UseAzureSignalR=true`) is what actually runs: it opens a `ServiceHubContext` via `Microsoft.Azure.SignalR.Management`, connects a client directly to the emulator (serverless clients bypass ASP.NET Core hubs — they negotiate a URL/token straight from the SignalR service), pushes one synthetic `QueueUpdated`, and logs whether the client received it. The app's real traffic — every Angular app, Blazor, all three frontends — keeps talking to the self-hosted `QueueHub` regardless of the flag; nothing about production behavior changes when the flag flips, only that this extra proof-of-round-trip runs at startup.

**Verified against a live emulator, not assumed:** `aspire run` with the flag on in both `AppHost/appsettings.json` and `ApiService/appsettings.json` starts an `mcr.microsoft.com/signalr/signalr-emulator` container; `GET /health` and `GET /queue` on the API stay unaffected (proving the flag doesn't touch the real request path); the emulator's own container logs show a clean connect/disconnect cycle for the demo's hub, with only the expected (and harmless) upstream-webhook connection-refused warning — the emulator tries to notify an Azure Functions-style webhook of connect/disconnect lifecycle events, which this POC doesn't run, but that path is unrelated to whether `Clients.All.SendAsync` messages reach an already-connected client. The send/receive mechanism itself was independently confirmed against the same running emulator container with a throwaway standalone client using the identical SDK calls.

**Found along the way — hub names can't contain hyphens:** Azure SignalR's hub-name validation only allows letters, digits, and `` _`,.[] ``. `negotiate`/connect silently accept a hyphenated hub name; the failure only surfaces as an opaque `400` from the REST send API the moment a broadcast is attempted. The first implementation used `queue-serverless-demo` and failed exactly this way; `AzureSignalRServerlessDemoService.DemoHubName` is `queueServerlessDemo` (camelCase) for this reason — worth flagging for the vendor team since it's not obvious from the SDK's negotiate-succeeds behavior.

**Rejected:** rerouting the app's real `QueueBroadcaster` traffic through `ServiceHubContext` when the flag is on. Serverless mode is a fundamentally different client topology (clients connect to the SignalR service directly, never to `QueueHub`) — swapping the app's actual frontends onto it would mean building and maintaining a second connection path in every Angular app and Blazor for a mode ADR-0001 didn't choose and this POC doesn't need working end-to-end. The flag stays a narrow, isolated proof of the emulator round-trip instead. Also rejected: falling back to the `Microsoft.Azure.SignalR.Emulator` dotnet tool as an AppHost executable resource (the fallback the issue anticipated in case Aspire lacked native support) — `Aspire.Hosting.Azure.SignalR` 13.4.6 (matching this repo's pinned Aspire version) has first-class `AddAzureSignalR(name, AzureSignalRServiceMode.Serverless).RunAsEmulator()` support, confirmed by reflecting on the restored package before writing any AppHost code, so the container-based resource (same shape as the Azurite emulator) was available directly.

## 2026-07-11 — Angular apps read their API address from a runtime-fetched `config.json`, not a build-time `environment.ts`

**Context:** issue #8 needed to decide, ahead of #12 actually containerizing the three Angular apps, how each app would learn the API's address. Angular's usual convention (`environment.ts`/`environment.prod.ts`, swapped by `fileReplacements` at build time) bakes that address into the compiled JS bundle.

**Decision:** each app fetches `/config.json` at startup, via an Angular app-initializer (`provideRuntimeConfig()` in `RuntimeConfigService`), before anything that needs the API address runs. Locally, each app ships a placeholder `public/config.json` pointing at the API's dev port (`http://localhost:5410`), copied verbatim into `dist/<app>/browser/config.json` by the Angular build.

**Why:** #12's plan (see that issue) has an nginx entrypoint script overwrite `config.json` from an environment variable at container *start* — after Aspire has assigned the API's actual address, which isn't known at `docker build` time. A build-time `environment.ts` value would freeze in an address before it exists, making the container image environment-specific (a fresh image per environment) instead of one image deployable anywhere with different runtime config — exactly the problem #12's task list calls out ("build-time baking would break service discovery").

**Rejected:** Angular's `fileReplacements`/`environment.ts` pattern — the standard approach for most Angular apps, wrong here specifically because of the container/service-discovery timing constraint above; picking it now would have meant redoing this in #12 anyway.

## 2026-07-11 — public-checkin: live position from the snapshot, token policy app-local, upload validation mirrored client-side

**Context:** #9 is the first real Angular UI on top of the #8 shared library — the visitor kiosk (check in → live "you are #N in line" → optional document upload). Three choices came up that the sibling apps (#10/#11) will inherit the pattern from.

**Decision 1 — position is derived from the `QueueHubService` snapshot, not the `POST /checkin` response.** `PositionView` recomputes the visitor's 1-based place among still-`Waiting` entries (ordered by parsed check-in time) from the authoritative snapshot on every live push / catch-up / poll. The check-in response's one-shot `position` is used only as a seed for the brief window before the snapshot first contains the new entry.

**Why:** the response position is correct only at the instant of check-in; people ahead get served while the visitor waits. Binding to the snapshot is what makes "position updates live when staff calls next" (the issue's acceptance criterion) fall out for free — and it's what exercises the shared reconnect/catch-up path from a real browser. Sorting locally (rather than trusting queue array order) is deliberate: a catch-up-*derived* snapshot doesn't guarantee server order (see `QueueHubService`'s remarks), so the component orders by `checkedInAt` itself.

**Decision 2 — the check-in-token flow lives in an app-local `KioskCheckInService`, not in the shared `QueueApiService`.** The shared client takes the token per call and stays out of *when* one is fetched or held (see its class doc). public-checkin's policy — fetch a fresh token immediately before each gated call (check-in, upload), never cache — is encoded once in this app-local service.

**Why:** how long a token is held is a UI-flow concern that differs per app (internal-queue #10 will hold a staff key for a whole session; public-checkin holds nothing). Server tokens are short-lived (5 min) and a kiosk sits idle between visitors, so a cached token risks being expired at use; re-fetching costs one cheap round-trip and removes a whole class of "expired mid-flow" failure. Putting this in `shared` would force one token policy on all three apps.

**Decision 3 — upload type/size validation is duplicated client-side, mirroring the server.** `DocumentUpload` carries the same allowlist (`application/pdf`, `image/jpeg`, `image/png`) and 10 MB cap as `DocumentEndpoints`, each constant naming the server symbol it shadows.

**Why:** the server stays the source of truth (it re-validates every upload); the client copy exists only so a visitor gets an instant, in-place rejection instead of picking a file, waiting for an upload, and getting a round-trip 400. The duplication is accepted (not abstracted into a shared constant) because the two live in different languages/deploy units and the server must own the real limit regardless — naming the shadowed symbol in a comment is the chosen "keep them in sync" mechanism, matching how the TypeScript `Contracts` mirrors are kept in step.

**Rejected:** trusting `POST /checkin`'s position for the whole session (goes stale immediately); a shared token-holding service (imposes one app's flow on all three); skipping client-side upload validation and relying on the server 400 (a worse kiosk experience for the common "wrong file" mistake).

## 2026-07-11 — One parameterized Dockerfile, `node:22-slim` + `nginx:alpine`, not three per-app files

**Context:** each Angular app needs its own container image, but all three share the exact same build (npm ci → build `shared` → build the app) and serve (nginx + an entrypoint script) recipe — only the app name and the assigned port differ.

**Decision:** a single `SignalRQueueDemo.Angular/Dockerfile`, parameterized by an `APP_NAME` build arg, builds all three images. `SignalRQueueDemo.AppHost/AppHost.cs` declares three `AddDockerfile` container resources against the same file, each passing a different `APP_NAME` via `WithBuildArg`. The build context is the whole `SignalRQueueDemo.Angular/` workspace folder, not a per-app subfolder, because every app's `tsconfig.json` path mapping resolves its `shared` import to `./dist/shared` — the workspace-root `package.json`/`angular.json` and the `shared` project must be in context regardless of which app is being built.

**Base images:** `node:22-slim` for the build stage (matches the workspace's Node 22 engine requirement with a smaller footprint than the full `node:22` tag; its size never reaches the final image since the build stage is discarded) and `nginx:alpine` for the serve stage (the image that actually ships — measured at ~93 MB total per app, ~26 MB unique beyond the shared `nginx:alpine`/`node:22-slim` base layers all three images share). `node:22-alpine` (musl libc) was considered for the build stage too, but some Angular/ng-packagr toolchain postinstall steps assume glibc and are less reliable on musl — a tradeoff worth making for the build stage (its size is irrelevant to the shipped image) but not for the serve stage, where `nginx` itself has no such postinstall scripts to trip over musl and the size savings matter.

**Rejected:** three near-duplicate Dockerfiles (one per app) — every line except the app name and a couple of comments would be identical, so a change to the build or serve recipe would need three synchronized edits instead of one.

## 2026-07-11 — CORS origins wired from AppHost via Aspire endpoint references, forced to `KnownNetworkIdentifiers.LocalhostNetwork`

> **Superseded 2026-07-12** — this approach injected `http://localhost:{proxyPort}` origins, which matched the *port* but not the *host* a real browser loads the apps from (Aspire serves them under `*.dev.localhost`, e.g. `http://public-checkin-signalrqueuedemo.dev.localhost:{port}`). CORS rejected every real-browser request even though same-origin `curl` checks passed. The AppHost CORS injection was removed and replaced by a loopback-origin allowance in the API — see the 2026-07-12 entry at the end of this file. The `API_BASE_URL` half of the wiring below (and its `LocalhostNetwork` reasoning) is unchanged and still correct; only the CORS half was wrong.

**Context:** `Cors:AllowedOrigins` in `SignalRQueueDemo.ApiService/appsettings.json` was seeded (see the 2026-07-11 CORS entry above) with the Angular CLI dev-server ports (`4200`–`4202`) as a placeholder, explicitly because the Angular apps weren't containerized yet and had no other origin to point at. Once containerized, each app is served from an Aspire-assigned `host:port` decided at `aspire run` time — the dev-server ports no longer describe where a browser actually loads these apps from, and neither does the container's internal Docker-published port (see below).

**Decision:** `AppHost.cs` overrides `Cors:AllowedOrigins[0..2]` on the `apiservice` resource with `Cors__AllowedOrigins__0/1/2` environment variables (Aspire's double-underscore config-binding convention), each set to one Angular container's `GetEndpoint("http", KnownNetworkIdentifiers.LocalhostNetwork)` reference — and the same explicit network identifier is used for the `API_BASE_URL` reference injected into each Angular container (see `AppHost.cs`'s comments at both call sites). This is a reference only — no `WithReference`/`WaitFor` from `apiService` to the Angular containers — because `apiService` only needs their endpoint *values*, not their running state, and the Angular containers already `WaitFor(apiService)`; making `apiService` also wait on them would deadlock every startup.

**Why `KnownNetworkIdentifiers.LocalhostNetwork` had to be explicit, found only by running `aspire run` and testing, not by reading the API surface alone:** `GetEndpoint(name)` with no network argument resolves an endpoint's value *relative to whichever resource ends up consuming it* (`EndpointReference`'s own remarks: "the reference will attempt to resolve itself based on the context of the requesting resource"). Consumed by a **container** (all three Angular resources), that produced `http://aspire.dev.internal:{port}` — Aspire's container-network tunnel hostname, resolvable only from inside Aspire's own Docker network (verified: reachable via `docker exec <container> wget ...`, unreachable via `curl`/`nslookup` from the host). Consumed by `apiService` (a plain host-process `ProjectResource`, not a container), the *same 2-argument call* on the Angular side would have resolved differently again. Neither is what either value is actually for: `API_BASE_URL` is read by browser JS (`RuntimeConfigService`) running on the visitor's/staff's own machine, and CORS's `Origin` header is set by that same browser to the address it loaded the page from — both need the one **host-reachable** URL, regardless of which Aspire resource happens to be the technical "consumer" of the reference. Passing `KnownNetworkIdentifiers.LocalhostNetwork` explicitly pins both directions to that one address, confirmed by re-running `aspire run` and checking a live container's `config.json` content and a CORS preflight (`curl -X OPTIONS .../queue -H "Origin: ..."`) both before (silently wrong: `aspire.dev.internal`, and a preflight with no `Access-Control-Allow-Origin` header) and after (correct, browser-usable `http://localhost:{port}`, preflight returns the header) the fix.

**A second, related trap worth flagging for the vendor team:** the `http://localhost:{port}` that `GetEndpoint(..., LocalhostNetwork)` resolves to for a container endpoint is **Aspire's own DCP proxy port**, not the port `docker ps` shows for that container. DCP fronts each container endpoint with a stable local proxy that forwards to whatever (ephemeral, different-every-run) port Docker actually published the container on internally — the proxy port is the one Aspire resolves in every reference expression, appears in the Aspire dashboard, and is what a browser must use; `docker ps`'s port column is an internal implementation detail, coincidentally also host-reachable, but a *different* port than the one every other Aspire-aware consumer (including this CORS wiring) agrees on. Confirmed by curling both — the `docker ps` port serves the app directly (Docker's own mapping) while the DCP-proxy port serves the same app *and* is the value found in every container's `config.json` and every CORS-allowed origin above; using the `docker ps` port for manual `curl` testing (as an early debugging pass here did) gives a misleading "CORS is broken" result purely from testing against the wrong port, not from wiring that is actually wrong.

**Rejected:** leaving `appsettings.json`'s dev-server-port placeholders as the only allowed origins post-containerization — would make every containerized Angular app's cross-origin API/hub calls fail CORS the moment `aspire run` assigns them different ports, silently breaking the exact acceptance criterion (live updates across all three frontends) the whole POC exists to demonstrate. Rejected relying on the network-less `GetEndpoint(name)` overload and hoping its ambient resolution happened to match — it does not, for exactly the reason above, and the failure is silent (a working `docker build`, a running container, a `config.json` that looks plausible) until someone opens the app in an actual browser or checks CORS with the *right* port.

**`docs/architecture.drawio` note:** the diagram already drew `public-checkin`/`internal-queue`/`queue-display` as "(Node build → nginx container)" boxes before this work — that shape didn't change, so the `.drawio` file itself needed no structural edit here; the new detail (which endpoint reference feeds `API_BASE_URL`/CORS, and the `LocalhostNetwork` pin) is the kind of wiring-level nuance the Mermaid diagram and this decisions log capture, not something a box-and-arrow diagram at this level of detail would show either way.

## 2026-07-12 — CORS accepts any loopback-family origin, replacing the AppHost origin injection

**Context:** the day-before approach (previous entry) injected each Angular container's origin into the API's `Cors:AllowedOrigins` as `http://localhost:{proxyPort}`, resolved via `GetEndpoint("http", KnownNetworkIdentifiers.LocalhostNetwork)`. In a real browser all three apps failed: `queue-display` stuck on "Loading queue…/Reconnecting…", `internal-queue` on "Reconnecting…", `public-checkin`'s check-in button reporting "Something went wrong checking you in." The API itself was reachable and healthy — the failure was CORS on every cross-origin call (REST *and* the SignalR hub's `negotiate`).

**Root cause (proved from the containers' own nginx access logs — the `Referer` header records the exact origin the browser loaded each page from):** Aspire does not serve these container endpoints at `http://localhost:{port}`. It serves them under its own `*.dev.localhost` hostname scheme — `http://public-checkin-signalrqueuedemo.dev.localhost:{proxyPort}`, `http://internal-queue-…`, `http://queue-display-…`. The injected allowlist entries had the right *port* but the host `localhost`, not `…dev.localhost`. CORS matches origins by exact `scheme://host:port`, so every request was refused. The earlier `curl` verification passed only because it sent `Origin: http://localhost:{port}` by hand — the one origin that was in the list and the one no browser actually sends. `*.dev.localhost` and `localhost` both resolve to 127.0.0.1, so the request still reached the API; only the `Origin` string differed, which is all CORS checks.

**Decision:** stop trying to enumerate the exact per-run origins from the AppHost. Instead the API's `KnownFrontends` policy accepts any origin whose host is loopback (`Uri.IsLoopback` — covers `localhost`/`127.0.0.1`/`::1` — plus any `*.localhost` subdomain, which is what matches Aspire's `*.dev.localhost` names), gated by `Cors:AllowLoopbackOrigins` (default `true`). Explicit `Cors:AllowedOrigins` entries still match exactly and are the *only* path used when the flag is off. The AppHost's `Cors__AllowedOrigins__0/1/2` injection was removed entirely; `API_BASE_URL` injection into the containers is unchanged. Verified against a live `aspire run` by replaying each app's real `*.dev.localhost` origin through the preflight, `/checkin/token`, a full `POST /checkin` (HTTP 200, entry created), the SignalR `negotiate`, and `GET /queue` — all now return the matching `Access-Control-Allow-Origin`; a non-loopback origin (`http://evil.example.com`) is still refused.

**Why this is acceptable here (and where it must not go):** every host in play resolves to loopback on the single, network-isolated machine this court POC runs on, so "trust the loopback family" is both correct for the environment and immune to Aspire reassigning ports/hostnames every run. CORS is not the trust boundary regardless — `StaffAuthFilter` and `CheckInTokenService` are, and they are unchanged. For any deployment reachable beyond localhost, set `Cors:AllowLoopbackOrigins=false` and populate `Cors:AllowedOrigins` with the real frontend origins; the code path for that is already there and tested by the negative case above.

**Rejected:** chasing the correct `*.dev.localhost` origin string from the AppHost (e.g. a different `KnownNetworkIdentifiers` value or string-building the hostname) — it re-introduces a dependency on Aspire's internal, undocumented, version-specific host-naming for a value that changes every run, to gain a stricter allowlist that buys nothing on a loopback-only machine. Also rejected: `AllowAnyOrigin()` — needlessly broader than "loopback", and it can't coexist with credentialed requests if those are ever added, whereas the loopback predicate can.

## 2026-07-12 — Staff sign-in verifies the key up front via `GET /staff/verify`

**Context:** the internal-queue sign-in screen originally stored any non-empty key and deferred validation to the first real staff action, which returned a 401. In practice that read as a bug: a mistyped key (easy, since the placeholder is a 51-character string and the field is masked) let you "sign in", show the queue (a public read), and then fail confusingly only when you clicked Call Next or opened a document — the last of which shows an inline "your staff key is invalid" rather than bouncing, so it looked like a broken feature, not a wrong key.

**Decision:** add a read-only, side-effect-free `GET /staff/verify` gated by the same `StaffAuthFilter` (204 on a valid key, the standard 401 ProblemDetails otherwise), and have the sign-in form `await` it before storing the key and handing off to the console. A 401 is surfaced inline as "that staff key is not valid"; any other failure gets a distinct "couldn't reach the server" message so a wrong key and an unreachable API aren't conflated. The console keeps its own 401 handling (bounce to sign-in on a mutation, inline message on a document read) as defence in depth for a key rotated mid-session — it's just no longer the primary way a wrong key surfaces. The sign-in screen also gained a hint naming `StaffAuth:Key` in `appsettings.json` as the source of the accepted value (the value itself isn't hardcoded in the UI, so rotating it doesn't strand the hint).

**Why a dedicated verify endpoint, not reuse of an existing one:** the mutating staff endpoints (`call-next`, `complete`) can't double as a credential check without a side effect, and the one read-only staff endpoint (`GET /queue/{id}/documents`) needs an entry id the sign-in screen doesn't have. A tiny purpose-built probe is clearer than contorting either. This does reverse the earlier "sign-in deliberately does no validation round-trip" stance recorded in the component's own comments — the round-trip is cheap for a once-per-session screen, and "the sign-in only accepts the right key" being literally true is worth it.

**Rejected:** a client-only length/format check (can't actually tell a valid key from an invalid one without the server); persisting the verified key so a refresh stays signed in (the in-memory-only policy is deliberate — see `staff-session.service.ts` — a shared staff terminal shouldn't stay authenticated across a refresh).

## 2026-07-12 — SignalR hub connection uses `withCredentials: false`

**Context:** after the loopback-CORS fix let the REST calls through, all three apps still sat in the "reconnecting…" state — data updated (via the polling fallback) but the live socket never connected. Symptom-wise identical to the CORS bug, but a different cause.

**Root cause:** `@microsoft/signalr` defaults `withCredentials` to `true` for every HTTP request it makes, negotiate included (confirmed in the client source: the option is coerced to `true` when left undefined). That makes negotiate a *credentialed* CORS request, and the browser only accepts a credentialed response if it carries `Access-Control-Allow-Credentials: true`. The API's CORS policy doesn't send that header (verified against the live negotiate response — `Access-Control-Allow-Origin` present, `Access-Control-Allow-Credentials` absent), so the browser blocked the negotiate response despite its HTTP 200, the hub connect failed, and `QueueHubService` fell back to polling. The REST calls were unaffected because Angular's `HttpClient` defaults `withCredentials` to `false`.

**Decision:** set `withCredentials: false` on the hub connection (`HubConnectionBuilder.withUrl(url, { withCredentials: false })`), matching the REST client. Nothing in this system authenticates with cookies — the staff key and check-in token are plain headers — so credentials on the socket were never needed; turning them off makes the hub's requests non-credentialed and the existing loopback CORS policy (which already works for REST) covers them too.

**Rejected:** adding `AllowCredentials()` to the API's CORS policy instead — it would also unblock the socket, but it grants credentialed cross-origin access nothing uses, and pairing `AllowCredentials` with the permissive loopback origin predicate is exactly the combination CORS guidance warns against. Fixing it on the one client that opts into credentials by default is narrower and truer to the design (header auth, no cookies).

## 2026-07-12 — Uploaded documents: blobs persist across restarts, and are deleted when the entry completes

**Context:** document *metadata* is stored in the queue's own store (SQLite by default, which persists to a file across `aspire run` restarts), but document *content* is stored in Azurite Blob Storage, whose emulator started empty every run. So a document uploaded in one run left a metadata row in the next run pointing at a blob that no longer existed — and viewing it failed with a 500 "content missing" that the staff UI surfaced as a confusing "failed to load document".

**Decision (two halves, chosen with the owner):** (1) give the Azurite emulator a data volume (`RunAsEmulator(e => e.WithDataVolume())`) so blob content has the same across-restart lifetime as the metadata; (2) delete an entry's documents — metadata rows *then* blob container — when it's completed (`HandleCompleteAsync`), so content lives exactly as long as the visitor is in the queue and no longer. Metadata is deleted before blobs so a partial failure can only orphan a (harmless) blob, never leave a metadata row pointing at deleted content — the same ordering rule the upload path already follows in reverse. The cleanup is best-effort and off the response path: the entry is already completed and broadcast, so a cleanup hiccup logs a warning rather than turning a successful complete into a retryable 500.

**Rejected:** making the SQLite metadata ephemeral instead (reset every run) — the owner wanted data to survive restarts, and the ADR treats SQLite as a real backend, so persisting both halves is the consistent choice. Also rejected: leaving the mismatch and only handling the missing blob gracefully — that turns "view my document" into a dead end, which defeats the feature.

## 2026-07-12 — `QueueEntry.DocumentCount` on the snapshot, refreshed live by broadcasting on upload

**Context:** the staff console should hide its "view documents" control for entries that have none, rather than making staff click into an empty list. That needs a per-entry document count wherever the console already reads entries — the live snapshot.

**Decision:** add a non-required `DocumentCount` (default 0) to `QueueEntry`, populated only on the snapshot path — `BuildSnapshotAsync` counts the documents store once for the whole queue (a single grouped query in SQLite; a client-side partition tally in Table Storage) rather than each entry row carrying a denormalized counter to keep in sync. Catch-up-replay entries leave it 0 and self-heal on the next full snapshot, matching how that path already treats its derived state as approximate. To make the count update the instant a waiting visitor attaches a file — not only on the next call-next/complete — the upload endpoint calls a new `IQueueRepository.RecordDocumentChangeAsync`, which appends a change event (so the sequence-numbered broadcast/catch-up protocol stays intact — no reused or skipped numbers) and pushes a fresh snapshot. The entry's lifecycle status is untouched; only its document set changed.

**Rejected:** a denormalized counter column on the entry row (another thing to keep in sync on every upload/delete); a separate "which entries have documents" endpoint the console polls (an extra round-trip and a second source of truth); broadcasting the upload by reusing the current sequence number (violates the strictly-increasing invariant the reconnect protocol depends on).

## 2026-07-12 — Public board shows masked names (first name + last initial), reversing the ticket-only stance

**Context:** the queue-display board originally showed ticket numbers only, with an explicit "never show names on a public display" constraint documented in `NowServing`/`WaitingList`. The owner asked for names so a visitor can recognise their own row at a glance.

**Decision:** show the name masked to first-name-plus-last-initial (`toPublicName`, e.g. "Jane Test" → "Jane T.") rather than the full name — a common waiting-room-board compromise that lets someone spot themselves without publishing full surnames (i.e. exactly who is at the court) to the whole room. The masking lives in one shared helper so it's trivial to change to full names or back to ticket-only. The internal staff console still shows full names — it's behind the staff-auth boundary, not public. Flagged to the owner as a reversal of a documented privacy constraint, with full names available on request.

## 2026-07-12 — public-checkin tracks multiple people per device; QR generated locally

**Context:** the kiosk originally showed one person's position and "check someone else in" *replaced* it — so checking in a family member lost your own spot. And the board needed a "how to check in" affordance.

**Decision:** the public-checkin shell now holds a *list* of check-ins and renders one live position card per person; "check someone else in" appends to the list (form reappears, existing cards stay), and a person leaves the list by completing (auto-reset) or "stop tracking". This directly fixes "I lost my spot" and lets one device watch several people at once. Separately, queue-display renders a check-in QR + URL; the URL comes from a new optional `RuntimeConfig.publicCheckinUrl` injected into only that container (AppHost → `PUBLIC_CHECKIN_URL` → `write-runtime-config.sh`), and the QR is encoded **locally** with the dependency-free `qrcode-generator` into an inline image data URL — never a QR web service, per the no-external-calls constraint. ## 2026-07-12 — Table Storage document count scoped to the current snapshot's entries, not the whole documents table

**Context:** a code review of commit `bdcb5cb` (GitHub issue #15, item #1) flagged `TableStorageQueueRepository.CountDocumentsByEntryAsync`: it walked the entire `QueueDocuments` table on every `BuildSnapshotAsync` — which runs on every mutation, every `GET /queue`, and every polling tick per connected client. Its cost was `O(all documents ever stored)`, unbounded relative to the queue size it was decorating, and made worse by document deletion on complete (`QueueEndpoints.HandleCompleteAsync`) being best-effort: a failed cleanup left orphaned rows that got rescanned forever after.

**Decision:** `CountDocumentsByEntryAsync` now takes the ids of the entries in the snapshot being built and issues one partition-scoped query per entry (`PartitionKey eq '{entryId}'`, run in parallel via `Task.WhenAll`) instead of one unscoped scan of the whole table. `DocumentTableEntity`'s PartitionKey is already the owning entry's id, so each query only ever touches that entry's own document rows.

**Why this is the right trade at this POC's scale:** it trades a single full-table scan for N small, partition-scoped round-trips, where N is the number of entries currently in the queue (dozens, per ADR-0001's stated volume) — not the number of documents ever uploaded across the system's lifetime. That is exactly the growth axis the review flagged, and scoping to the snapshot's own entries removes it: cost now tracks queue size, the same axis SQLite's per-snapshot `GROUP BY` already tracks (which the review found acceptable). It does not reclaim orphaned rows left behind by a failed best-effort delete — those remain dead storage — it only stops paying to scan them on every request.

**Rejected:** caching/denormalizing the count onto the entry row — reintroduces exactly the "keep a counter in sync on every upload/delete" problem the original `QueueEntry.DocumentCount` decision (above) deliberately avoided by counting from the documents store instead. Rejected leaving it as a documented limitation — unlike the DocumentCount catch-up gap below, the fix here is a small, self-contained change with no cross-layer schema implications, so there was no real reason not to just fix it.

## 2026-07-12 — Catch-up `DocumentCount=0` gap: documented as an accepted POC limitation, not fixed

**Context:** the same review (issue #15, item #3) noted that `GET /queue/since/{seq}` catch-up replays entries from the change-event log, which carries an entry's lifecycle fields but not its document count (see the `QueueEntry.DocumentCount` decision above — that field is populated only on the snapshot path). If an entry that already has documents changes lifecycle state (e.g. gets called next) while a staff console is briefly disconnected, the catch-up diff replays it with `DocumentCount=0`, and the console's "View documents" control stays hidden for that entry until the next full snapshot arrives.

**Decision:** left as-is, documented rather than fixed — expanded the XML doc on `QueueEntry.DocumentCount` (`SignalRQueueDemo.Contracts/QueueDomainTypes.cs`) to name the concrete symptom (a hidden "View documents" control) instead of only the general "self-heals" framing.

**Why not fixed:** a real fix means carrying the document count through the change-event log itself — a schema change to `QueueChangeEventTableEntity`/the SQLite change-event table, backfill/versioning considerations for existing local data, and matching updates to both repository implementations' catch-up paths — for a gap that is narrow (only manifests for an entry that both has documents *and* changes lifecycle state *and* the console is disconnected at that exact moment) and self-correcting (heals on the very next full snapshot, which every reconnect and every 5s poll eventually triggers). That's disproportionate machinery for a POC reference implementation versus the Table Storage scan fix above, which was a small, local change with no schema impact.

**Rejected:** the schema-widening fix described above — correct for a production system where this control staying briefly hidden is a real support complaint, out of proportion here. Also rejected: silently leaving it undocumented — CLAUDE.md's documentation standard requires every non-obvious gap to say why it exists, not just that it does.

## 2026-07-12 — `POST /queue/{id}/cancel`: a public, check-in-token-gated write; `Cancelled` is its own terminal state, counted as neither completed nor active

**Context:** the kiosk's "Stop tracking" button previously only dropped the position card locally — the entry stayed `Waiting`/`Serving` in the staff queue forever, so a visitor who left still appeared to be in line. A visitor abandoning the queue needs to actually leave it. This added a `Cancelled` status (`QueueStatus`, `SignalRQueueDemo.Contracts/QueueDomainTypes.cs`), a `CancelAsync` on `IQueueRepository` (both implementations), and `POST /queue/{id}/cancel`, wired to the kiosk button.

**Decision — cancel is gated by `CheckInTokenFilter`, the same public-write gate as `POST /checkin`, not `StaffAuthFilter`.** A visitor tapping "Stop tracking" isn't staff, so the staff-key gate the *staff-initiated* `complete`/`call-next` use is wrong here — but the endpoint must not be unauthenticated either. Cancelling deletes the entry's documents and blobs (same best-effort cleanup as complete), so an open, id-only endpoint would let any caller destroy an arbitrary entry's data by guessing/harvesting ids (`GET /queue` returns them). It therefore carries the short-lived check-in token, exactly the "protect writes, not reads" anti-forgery pattern the rest of the public path already uses; the kiosk's app-local `KioskCheckInService.cancel` fetches a fresh token immediately before the call, identical to check-in and upload. `CancelAsync` also rejects an already-`Completed`/`Cancelled` entry (409), so a double-tap is a no-op, not a duplicate broadcast.

**Decision — `Cancelled` is a distinct terminal state and is counted in *no* summary total, not folded into `TotalCompleted`.** An earlier cut of this work added `|| Cancelled` to the `TotalCompleted` count in both repositories. That conflated two different outcomes ("we served this person" vs. "this person left") under one label, and — worse — disagreed with the client: `QueueHubService.deriveSnapshotFromIndex` (the reconnect catch-up path) counts only `Completed`, so the same client would show a different completed count depending on whether a total came from the server or was re-derived locally after a catch-up. Reverting the server to count only `Completed` makes the two agree by construction and keeps the stat honest; a cancelled entry simply drops out of the active `Waiting`/`Serving` counts and displays (exactly as a completed one does), which is all any UI needs. The TypeScript `QueueStatus` mirror (`shared/.../queue.models.ts`) gained `Cancelled = 3` in the same change — required by that file's own "must stay in the exact declaration order of the C# enum" rule, and the reason the client can now recognise the state at all.

**Decision — the kiosk emits "finished" immediately, then fires the backend cancel best-effort.** `PositionView.startOver` drops the card locally first and only then calls `cancel` fire-and-forget, so the local reset never waits on (or is blocked by) the round-trip, and — unlike the first cut, which emitted `finished` from inside the HTTP callback — nothing touches the component after it has been torn down. A failed cancel is swallowed: the card is already gone locally, and the entry harmlessly stays in the staff queue (the same "don't strand the visitor on an API failure" intent as before).

**Table Storage counter note:** `CancelAsync` decrements the `WaitingCount` position counter only when the cancelled entry was `Waiting` (mirroring `CallNextAsync`'s `Waiting → Serving` decrement); a `Serving → Cancelled` transition leaves it untouched, exactly as `CompleteAsync`'s `Serving → Completed` does. It uses the same optimistic-concurrency retry loop (ETag + `412` retry) as the other entry-row mutations.

**Rejected:** an unauthenticated cancel endpoint (any caller could destroy an arbitrary entry's documents by id); gating it with `StaffAuthFilter` (a public kiosk visitor has no staff key); counting `Cancelled` as `Completed` (mislabels the stat and desyncs the client's locally-derived total); adding a separate `TotalCancelled` field (speculative — no UI consumes even `TotalCompleted` today, so a new field would be unused surface, against CLAUDE.md's "nothing speculative").

## 2026-07-12 — `SignalRQueueDemo.Contracts` merges into `SignalRQueueDemo.Shared` alongside the persistence extraction

**Context:** issue #13 (Blazor experiences) needed `IQueueRepository`/`IDocumentRepository`/`DocumentBlobStore` extracted out of `SignalRQueueDemo.ApiService` into something `SignalRQueueDemo.Web` could reference too — already flagged as a prerequisite by the "Blazor is self-encapsulated" entry below. The natural default was a second new project (`SignalRQueueDemo.QueueService`) sitting alongside the existing `SignalRQueueDemo.Contracts`. The owner redirected: fold `Contracts` into the new project instead, so both hosts take one shared reference, not two.

**Decision:** one project, `SignalRQueueDemo.Shared`, replacing `SignalRQueueDemo.Contracts` (deleted). It holds four things: `Contracts/` (the DTOs, namespace unchanged at `SignalRQueueDemo.Contracts` so no consuming `using` needed to move), `Persistence/` (`IQueueRepository`/`IDocumentRepository` and both backends, moved verbatim from `ApiService`, namespace changed to `SignalRQueueDemo.Shared.Persistence*`), `Auth/StaffKeyVerifier` (lifted out of `StaffAuthFilter` so Blazor's sign-in page uses the identical fixed-time comparison), and `Documents/DocumentUploadService` (the upload validation/write logic lifted out of `DocumentEndpoints.HandleUploadDocumentAsync`, so Blazor's own upload component enforces the exact same content-type allowlist/size cap/per-entry cap via the same constants, not a hand-copied second set of numbers). `QueueServiceCollectionExtensions.AddQueueService()` is the one `Persistence:Provider` registration switch both `ApiService` and `Web`'s `Program.cs` call.

**Why one project, not two:** `ApiService` and `Web` needed both the DTOs and the persistence/auth code regardless — a second project would have meant every future consumer taking two `ProjectReference`s for what is, from either host's perspective, one coherent "everything that isn't process-specific" dependency. Merging them means a change to a DTO or a repository method signature is a compile error in both hosts from one edit, not a coordination exercise across two projects that happen to always change together.

**Rejected:** keeping `Contracts` separate and adding a new `QueueService` project referencing it (the original plan) — no correctness difference, just one extra project and one extra `ProjectReference` per host for no isolation benefit, since nothing in this solution consumes `Contracts` without also needing the persistence code (the Angular workspace mirrors the DTO *shapes* by hand in TypeScript; it never references the C# project at all, so `Contracts`' physical location is invisible to it either way).

## 2026-07-12 — `QueueHub.NotifyMutation`: re-derive the broadcast from the repository, never trust the caller's payload

**Context:** `QueueHub` was documented and built with zero client-callable methods — every write went through `ApiService`'s REST endpoints, which already build the `QueueUpdated` broadcast from committed repository state and hand it to `QueueBroadcaster`. Once `SignalRQueueDemo.Web` writes to `IQueueRepository` directly (see the entries above), that write never reaches `QueueBroadcaster` on its own — Blazor and `ApiService` are separate Aspire processes, and only `ApiService` hosts the hub. Without a way to tell the hub "something changed," a Blazor-originated check-in/call-next/complete would update the writer's own circuit but never reach any other connected client (Angular, another Blazor tab).

**Decision:** `QueueHub` gains exactly one client-callable method, `NotifyMutation(long sequenceNumber)`. It takes **only** the sequence number — never a client-supplied `QueueUpdated` — and re-reads the actual change from `IQueueRepository.GetChangesSinceAsync(sequenceNumber - 1)` plus a fresh `GetStateAsync()` for the summary, before handing the reconstructed `QueueUpdated` to the same `QueueBroadcaster.BroadcastAsync` every REST-triggered broadcast already uses. If the requested sequence number can't be matched to exactly one change (already superseded, or the diff fell back to a snapshot), it silently does nothing — the caller's own circuit already has correct state from the write it just made, and every other client picks the change up on its own next mutation-triggered broadcast or reconnect catch-up.

**Why not accept the caller's own `QueueUpdated`:** `QueueHub`'s hub endpoint (`/hubs/queue`) is reachable by any SignalR client that can open a connection to it. `RequireCors(CorsPolicies.KnownFrontends)` stops a *browser* from connecting cross-origin, but CORS is a browser-enforced concept — it does nothing against a non-browser caller (`curl`, a hand-rolled SignalR client) invoking a hub method directly. A `NotifyMutation` that trusted and rebroadcast a caller-supplied payload would let anyone connected to the hub spoof "Now Serving" or a fabricated queue state on every connected display and staff console — a real integrity problem for a public courthouse display, not a hypothetical one. Re-deriving from the repository closes that off entirely: the worst a hostile caller can do is trigger a redundant broadcast of data that's already public via the unauthenticated `GET /queue/since/{seq}` endpoint. This preserves the hub's original "never decides what changed" invariant rather than contradicting it — the one client-callable method still only ever repeats what the repository says happened.

**Rejected:** `NotifyMutation(QueueUpdated update)` accepting the ready-built payload Blazor's own repository call already returns (simplest to implement, and *would* work correctly for a well-behaved Blazor client — but trusts every other caller on the same public hub endpoint equally, which is the integrity problem above). Also rejected: gating `NotifyMutation` behind the staff key or check-in token — it never exposes anything the unauthenticated `GET /queue/since/{seq}` doesn't already, so there is nothing here for either gate to protect.

## 2026-07-12 — AppHost gap: `webfrontend` needed the same Azurite/`QueueDb` wiring as `apiservice`, including an absolute path

**Context:** found while implementing #13, not anticipated by the original design — `SignalRQueueDemo.AppHost` wired the Azurite `tables`/`blobs` resources only to `apiservice`. Once Blazor calls `IQueueRepository`/`IDocumentRepository`/`DocumentBlobStore` directly, `webfrontend` needs identical access to whichever store is active. Worse, for the default SQLite provider: `ConnectionStrings:QueueDb` in each project's own `appsettings.json` is `Data Source=App_Data/queue.db`, a path relative to the **process's own working directory** — which `aspire run` sets to each project's own folder. Left alone, `ApiService` and `Web` would each quietly open a *different* physical `.db` file, so a Blazor check-in would simply never show up anywhere else — no error, no exception, just a queue that silently forked in two.

**Decision:** `AppHost.cs` now gives `webfrontend` the same `.WithReference(tables).WaitFor(tables).WithReference(blobs).WaitFor(blobs)` `apiservice` already has, and computes one absolute `QueueDb` connection string — `Path.GetFullPath(Path.Combine(builder.AppHostDirectory, "..", "SignalRQueueDemo.ApiService", "App_Data", "queue.db"))` — injected into **both** resources via `WithEnvironment("ConnectionStrings__QueueDb", ...)`. `builder.AppHostDirectory`-relative paths are already the established pattern in this file (the Angular `AddDockerfile` build context uses the same idiom), so this isn't a new one. Each project's own `appsettings.json` keeps the plain relative connection string as a fallback for a bare `dotnet run --project <ApiService|Web>` outside Aspire — that path only ever exercises one process against its own local file anyway, so the two-processes-diverge failure mode doesn't apply there.

**Why this belongs in decisions.md as a caught gap, not silently fixed:** the original "Blazor is self-encapsulated" design (below) named the persistence-extraction prerequisite but didn't anticipate that direct repository access also requires *infrastructure* parity between the two hosts, not just a code reference. Recording it here is the honest version of "we found this while building it" the vendor team should see, not a retroactively-tidied plan that implies it was foreseen.

**Rejected:** a shared, hardcoded absolute path in each project's own `appsettings.json` (works, but re-introduces a machine-specific absolute path into source-controlled config, and two independently-edited copies can still drift); an environment variable set outside Aspire entirely, e.g. in a `.env` file (works for `aspire run` today but abandons Aspire's own resource-reference model for no benefit, and doesn't help the standalone-`dotnet run` fallback case at all).

## 2026-07-12 — Blazor's check-in path skips the HMAC check-in token; the interactive circuit is the anti-abuse boundary instead

**Context:** `POST /checkin`'s REST path is gated by a short-lived, stateless HMAC token (`CheckInTokenService`/`CheckInTokenFilter`) — see the check-in token entries elsewhere in this file — specifically because a public kiosk SPA on a different origin can't be assumed to carry a same-site cookie, and a stateless token was judged the least-bad anti-abuse measure available to it. Blazor's `/checkin` page calls `IQueueRepository.CheckInAsync` directly, bypassing `QueueEndpoints`/`CheckInTokenFilter` entirely (per "Blazor is self-encapsulated," below) — so the question is whether it needs an equivalent gate of its own.

**Decision:** no — Blazor's `/checkin` page does not implement or call the check-in token scheme at all. The reasoning the token exists for (a public SPA "fundamentally cannot hold a secret a determined attacker can't also read," per `CheckInTokenService`'s own remarks) doesn't apply to a Blazor Server interactive circuit: the check-in form only renders and only submits over an already-established, per-circuit SignalR connection between the browser and this specific server process, carrying `Microsoft.AspNetCore.Components.Server`'s own circuit-id/antiforgery protections (`app.UseAntiforgery()` is already registered in `Web/Program.cs`). A drive-by script cannot forge a check-in without first establishing a real, stateful circuit the same way a real browser tab does — a materially stronger bar than "POST a JSON body to a public REST endpoint with a replayable bearer token," which is exactly the threat model the HMAC token was built to raise the bar against.

**Why not port the token scheme to Blazor anyway, for symmetry:** the token exists to compensate for a REST endpoint's statelessness — no session, no origin check beyond CORS, nothing tying a request to a specific prior page load. Blazor's request model doesn't have that gap: the circuit itself already provides what the token approximates. Adding it anyway would be ceremony copied from a different trust model, not a measurable security improvement — CLAUDE.md's standard is to document *why* a thing exists or doesn't, and "we ported it without checking whether the threat it defends against is even present" is not a why.

**Rejected:** implementing a Blazor-side token scheme for exact 1:1 symmetry with the Angular path — see above. Also rejected: leaving this undecided/undocumented, since a reviewer skimming `CheckIn.razor` and *not* seeing a token check could easily mistake it for an oversight rather than a considered call.

## 2026-07-12 — Document viewing on Blazor: a local token-gated streaming endpoint, not a call to `ApiService`'s REST endpoint

**Context:** issue #13 explicitly left this as an open choice: "document viewing on staff — those still call the API endpoints for blob access, or use a shared service; record the choice." `ApiService`'s `GET /queue/{id}/documents/{docId}` streams a document back gated by the `X-Staff-Key` header. Blazor's `DocumentViewer` needs to put a document's bytes into an `<iframe>`/`<img>` `src` or a download link — none of which can attach a custom header; the browser only ever sends a bare `GET` to whatever URL is in the markup.

**Decision:** `SignalRQueueDemo.Web` hosts its own local minimal-API endpoint, `GET /staff/documents/{entryId}/{docId}`, calling `IDocumentRepository`/`DocumentBlobStore` directly (the same calls `DocumentEndpoints.HandleGetDocumentAsync` makes) — never proxying to `ApiService` over HTTP. It's gated by a fresh, short-lived (2 minute), per-document HMAC token (`DocumentAccessTokenService`, signed with the same `StaffAuth:Key` already required in `Web`'s config) that `DocumentViewer.razor` mints server-side on every render and embeds in the URL, instead of a header. The signature covers `entryId`+`documentId`, so a token minted for one document can't be replayed against another.

**Why local-endpoint-plus-token, not a REST call to `ApiService`:** calling `ApiService`'s own document endpoint would need an `HttpClient`, would need to forward the `X-Staff-Key` header (or mint yet another credential), and would directly contradict "Blazor is self-encapsulated" (no REST calls to the API) for the one piece of this feature that happens to need a real URL. Since the shared library already gives Blazor direct repository/blob access, adding a *local* endpoint that calls the same shared code `ApiService`'s endpoint calls keeps that principle intact — the only thing that changed is that HTML, not `IQueueRepository`, is what forced a URL to exist at all.

**Why reuse `StaffAuth:Key` as the token's signing key rather than a new secret:** this token never crosses the staff trust boundary that key already gates — only markup already gated by `StaffSessionService.IsSignedIn` ever calls `Issue()`. It doesn't need independent rotation from the staff key, so introducing a second secret to provision and document would be duplication without a corresponding security benefit.

**Rejected:** a generic, reusable HMAC-token helper shared between `CheckInTokenService` (ApiService) and this new service (Web) — both are ~30-line signed-timestamp validators, but factoring one abstraction for two call sites with different scoping needs (a bare expiry vs. an expiry plus two resource ids) was judged premature; revisit if a third token use case appears. Also rejected: embedding the `X-Staff-Key` value itself in the URL as a query parameter instead of a purpose-built token — it would work, but leaks the actual staff credential into browser history/server access logs instead of a single-purpose, narrowly-scoped, short-lived value.

## 2026-07-13 — Post-#13 code review fixes: `IDbContextFactory`, `QueueCompletionService`, and locking `QueueRealtimeService`'s shared state

**Context:** a code review of the #13 diff (Blazor experiences) surfaced six findings, all in the new code. Two were the same root cause; the rest were independent. Documenting the fixes together since they all landed as one follow-up pass, not because they're one decision.

**1. Completing an entry via Blazor's staff console never deleted its documents.** `QueueEndpoints.HandleCompleteAsync` (REST) deletes an entry's documents on complete; `QueueConsole.razor`'s `HandleCompleteAsync` (Blazor, calling `IQueueRepository.CompleteAsync` directly) never did, since that cleanup lived inline in the REST handler, not the repository. **Decision:** extracted `QueueCompletionService` (`SignalRQueueDemo.Shared/Documents/QueueCompletionService.cs`) — completes the entry then deletes its documents/blobs, returning both the completion `Update` and an optional documents-cleared `Update` for the caller to broadcast (same "service does the work, caller decides how to broadcast" split as `DocumentUploadService`). Both `QueueEndpoints.HandleCompleteAsync` and `QueueConsole.HandleCompleteAsync` now call it. **Rejected:** duplicating the delete calls into `QueueConsole.razor` directly — exactly the "two hand-kept-in-sync copies" problem `DocumentUploadService` was extracted to avoid for uploads; the same reasoning applies to completion.

**2 & 3. `SqliteQueueRepository`/`SqliteDocumentRepository` held a single injected `QueueDbContext`, shared for the repository's entire DI-scope lifetime.** In ApiService that scope is one HTTP request — harmless. In Blazor, `IQueueRepository` is resolved once per **circuit** (a whole browser-tab session) and called concurrently from the circuit's renderer thread (button clicks), `QueueRealtimeService`'s poll `Timer` (a threadpool thread), and its reconnect-triggered catch-up (another threadpool thread). A single long-lived `DbContext` is not thread-safe (concurrent use throws `InvalidOperationException`) and, as a tracking context, kept returning the first-loaded copy of each entity instead of picking up writes ApiService's separate process made to the same database — silently defeating the reconnect/catch-up guarantee for Blazor specifically. **Decision:** switched `AddDbContext<QueueDbContext>` to `AddDbContextFactory<QueueDbContext>` in `QueueServiceCollectionExtensions.AddQueueService`; both SQLite repositories now take `IDbContextFactory<QueueDbContext>` and create a short-lived context per method call (`await using QueueDbContext dbContext = await this.dbContextFactory.CreateDbContextAsync(ct);`) instead of holding one. Fixes both processes identically — ApiService already got a fresh context per request via its own scope, so this is a no-op there, but it removes the sharp edge for any future direct-repository-access consumer, not just Blazor. **Rejected:** giving `QueueRealtimeService`/`QueueConsole`/etc. their own `IServiceScopeFactory`-created scope per operation instead — works, but pushes the "don't share a DbContext across threads/time" responsibility onto every future caller instead of fixing it once where the repository is implemented; `IDbContextFactory` is also the pattern Microsoft's own Blazor Server + EF Core guidance recommends for exactly this scenario.

**4. `DocumentViewer` closed whatever document a staff member was reading on every unrelated queue update.** `OnParametersSetAsync` reset `selectedDocumentId` and refetched the document list on *every* call, but Blazor calls it on every parent re-render, not just when `EntryId` changes — and `QueueConsole` re-renders on every `QueueUpdated` (any check-in, call-next, anywhere). **Decision:** track the last-loaded `EntryId` in a field and skip the reset/refetch when it's unchanged.

**5. `QueueRealtimeService`'s `entriesById` (a plain `Dictionary`) and `highestSequenceSeen` (a bare `long`) were mutated from multiple threads with no synchronization** — the same thread mix as finding 2/3 (hub-callback thread, poll timer, circuit thread via `PublishAsync`). **Decision:** added a `Lock stateLock` field; every read/write of both goes through it (`GetHighestSequenceSeen`/`UpdateHighestSequenceSeen` helpers, and `lock` blocks in `ApplySnapshot`/`DeriveSnapshotFromIndex`/`CatchUpAsync`'s patch loop). `Changed` is still raised outside the lock so a subscriber's re-render can't deadlock against it or hold the lock for the duration of arbitrary event-handler code.

**6. `HandleLiveUpdate` raised `Changed` twice per live push** — once inside `ApplySnapshot`, once again right after setting `LastUpdate` — so every subscribed component rendered twice per queue update. **Decision:** set `LastUpdate` before calling `ApplySnapshot` (which already raises `Changed` once) and dropped the redundant second call.

## 2026-07-13 — Merging #13 with #16's cancel feature: `QueueCancellationService`, and wiring Blazor's "Stop tracking" to actually cancel

**Context:** #13 (this Blazor work) and #16 (the cancel-queue feature, `POST /queue/{id}/cancel`) were developed in parallel and landed as a real merge, not a fast-forward — both touched `SqliteQueueRepository`/`TableStorageQueueRepository`/`IQueueRepository`/`QueueEndpoints.cs`. The merge itself was mechanical (git's rename-aware merge placed #16's `CancelAsync` additions correctly against #13's file moves/`IDbContextFactory` rewrite with only one textual conflict, in `SqliteQueueRepository.cs`, resolved by porting `CancelAsync` into the `IDbContextFactory` shape the other methods already use). Two things needed real reconciliation, not just conflict-marker resolution:

**1. `QueueEndpoints.HandleCancelAsync` still had the old inline document-cleanup pattern** — the same one `HandleCompleteAsync` had before finding 1 (above) extracted `QueueCompletionService`. Left alone, this would have reintroduced finding 1's exact bug class for cancel: a Blazor-originated cancel would leave documents behind, since only the REST handler's inline `try`/`catch` deleted them. **Decision:** extracted `QueueCancellationService` (`SignalRQueueDemo.Shared/Documents/QueueCancellationService.cs`), a straight mirror of `QueueCompletionService` — cancels the entry via `IQueueRepository.CancelAsync`, then deletes its documents/blobs best-effort, returning both updates for the caller to broadcast. `QueueEndpoints.HandleCancelAsync` now calls it instead of the inline duplicate.

**2. Blazor's own `PositionView`'s "Stop tracking" button never called cancel at all** — it only removed the card from local tracking (`OnFinished` → the parent drops it from its dictionary), because it was written before #16's cancel feature existed on this branch. Left as-is, a Blazor visitor tapping "Stop tracking" would vanish from their own kiosk screen while silently remaining `Waiting`/`Serving` in the staff queue forever — worse than the bug #16 was built to fix, and inconsistent with the now-merged Angular kiosk (`PositionView.startOver`) sitting right next to it in the same repo. **Decision:** ported `startOver`'s exact semantics — drop the card locally first (`await OnFinished.InvokeAsync()`, so the local UX never waits on the backend), then fire a best-effort `QueueCancellationService.CancelAsync` call **only if the entry was still `Waiting` or `Serving`** at the moment of stopping (never for an already-`Completed` entry, which the auto-reset countdown path never touches either), swallowing a failed cancel exactly like Angular does — the entry simply stays in the staff queue rather than the visitor being blocked or shown an error for a card that's already gone from their own screen. The successful result's update(s) are published via `QueueRealtimeService.PublishAsync` so every other connected client sees the cancellation, same as every other Blazor-originated write in this codebase.

**Rejected:** leaving Blazor's "Stop tracking" as local-only and treating the parity gap as out of scope for this merge — a silent behavioral regression relative to the Angular app it sits beside is exactly the kind of thing a careless merge ships and a careful one doesn't; CLAUDE.md's "vendor team reads this code" bar means the two frontend stacks should behave the same for the same user action unless a documented reason says otherwise.

## 2026-07-13 — #17: server-side QR generation, and injecting `PublicCheckinUrl` into two resources

**Context:** the "check in from your phone" call-to-action on the queue-display board needs a QR image of the public-checkin URL. It was originally encoded in the browser by the Angular `qrcode-generator` dependency into an inline data URL.

**Decision:** moved QR encoding **server-side** into a shared `QrCodeHelper.GenerateSvg` (`SignalRQueueDemo.Shared`, backed by `Net.Codecrete.QrCodeGenerator`), exposed to the Angular board through a new `GET /checkin/qr` endpoint that returns an `image/svg+xml`. The board now points an `<img src>` at that endpoint instead of carrying a QR library; `qrcode-generator` / `@types/qrcode-generator` were dropped from `package.json`. The Blazor board's own `CheckInQr.razor` calls `QrCodeHelper` **directly in-process** (not over HTTP), consistent with the standing "Blazor is self-encapsulated" principle — one encoder, two callers, no client-side QR code anywhere.

**Why still no external call:** the SVG is generated on the one isolated machine and never fetched from a QR web service, so the court's no-external-calls constraint holds exactly as it did with the client-side encoder; the change is purely *where* the bytes are produced. The QR endpoint reads its URL from the API's own `PublicCheckinUrl` config (not from a query parameter) so a caller can't make the API encode an arbitrary URL.

**AppHost wiring — and a bug fixed in review:** `publicCheckin` is declared *after* both `apiService` and `webfrontend` in `AppHost.cs`, so its endpoint can only be referenced further down the file. The API and Blazor each get `PublicCheckinUrl` injected there via `WithEnvironment`. An initial version looked `webfrontend` up out of `builder.Resources` and cast it to `IResourceBuilder<ProjectResource>` — that collection holds `IResource` instances, not resource *builders*, so the cast threw `InvalidCastException` and crashed `aspire run` at startup. **Fixed** by capturing the `webfrontend` builder in a variable at its declaration and calling `.WithEnvironment` on it directly (no lookup, no cast, no null check).

**Rejected:** having Blazor's `CheckInQr` fetch `GET /checkin/qr` from the API like Angular does — it would reintroduce a REST call from Blazor to the API that "Blazor is self-encapsulated" exists to avoid, when the same shared helper is already callable in-process.

## 2026-07-13 — #17: dark-themed, framework-less CSS via Blazor CSS isolation

**Context:** the Blazor experiences were using default Bootstrap scaffolding (sidebar `NavMenu`, `top-row`, inline styles).

**Decision:** applied a shared dark theme (`wwwroot/theme.css`, linked from `App.razor`) plus per-component `.razor.css` isolation files, and stripped `MainLayout` down to a full-bleed `<main>` (no sidebar/nav) so each experience renders kiosk/board full-screen.
